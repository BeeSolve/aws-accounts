# Requirements Document

## Introduction

Remote Execution V2 adds a `remote` command group to the `@beesolve/aws-accounts` CLI that enables executing AWS operations via a Lambda function while keeping plan computation local. The architecture separates concerns: the CLI owns configuration parsing and diffing (plan), while a Lambda function owns AWS SDK execution (scan, apply) and state persistence on S3. This enables running the tool from environments with minimal IAM permissions (only `lambda:InvokeFunction` after bootstrap).

## Glossary

- **CLI**: The `@beesolve/aws-accounts` command-line tool running on the user's machine
- **Lambda_Handler**: The single AWS Lambda function file that routes on an `action` field in the event payload and executes AWS SDK operations
- **State_Bucket**: The S3 bucket created during bootstrap that stores the canonical `state.json`
- **State_Cache**: A local copy of `state.json` stored on the user's machine with a configurable TTL for fast plan computation
- **Context_File**: The `aws.context.json` file that stores Lambda ARN, state cache TTL, and existing organizational metadata
- **Operation**: A single planned change (e.g., moveAccount, createOu) as defined in the existing `operations.ts` schema
- **Plan**: The ordered list of Operations produced by diffing desired state against current state
- **Pre_Signed_URL**: A time-limited S3 URL generated by the Lambda_Handler for the CLI to download state
- **Bootstrap**: The process of deploying Lambda_Handler, State_Bucket, and IAM execution role using raw AWS SDK calls
- **State_Lock**: The combination of S3 conditional writes and Lambda reserved concurrency of 1 that prevents concurrent applies
- **Lambda_Zip**: The pre-built deployment artifact produced by the `build:lambda` script

## Requirements

### Requirement 1: Remote Bootstrap Deployment

**User Story:** As a DevOps engineer, I want to deploy the remote execution infrastructure with a single command, so that I can start using remote execution without manual AWS Console setup.

#### Acceptance Criteria

1. WHEN the user runs `aws-accounts remote bootstrap`, THE CLI SHALL create an S3 bucket with a deterministic name derived from the AWS account ID and region
2. WHEN the user runs `aws-accounts remote bootstrap`, THE CLI SHALL create an IAM execution role with permissions for Organizations, IAM Identity Center, S3 (state bucket only), and CloudWatch Logs
3. WHEN the user runs `aws-accounts remote bootstrap`, THE CLI SHALL deploy a Lambda function using the pre-built `dist/lambda.zip` artifact with the IAM execution role attached
4. WHEN the user runs `aws-accounts remote bootstrap`, THE CLI SHALL set the Lambda reserved concurrency to 1
5. WHEN bootstrap completes successfully, THE CLI SHALL persist the Lambda ARN and S3 bucket name in the Context_File under the `deployment` key
6. WHEN bootstrap completes successfully, THE CLI SHALL persist a default state cache TTL value in the Context_File
7. IF the `dist/lambda.zip` file does not exist, THEN THE CLI SHALL exit with an error instructing the user to run `npm run build:lambda` first
8. IF any AWS resource already exists during bootstrap, THEN THE CLI SHALL reuse the existing resource and log that it was reused rather than created
9. WHEN the user runs `aws-accounts remote bootstrap`, THE CLI SHALL use raw AWS SDK calls without CloudFormation

### Requirement 2: Lambda Build Script

**User Story:** As a developer, I want a build script that produces the Lambda deployment artifact, so that the bootstrap command has a ready-to-deploy zip file.

#### Acceptance Criteria

1. WHEN the user runs `npm run build:lambda`, THE Build_Script SHALL bundle the Lambda_Handler source into a single ESM file using esbuild
2. WHEN the user runs `npm run build:lambda`, THE Build_Script SHALL produce a zip file at `dist/lambda.zip` containing the bundled handler
3. THE Build_Script SHALL target Node.js 24 runtime and ESM module format
4. THE Build_Script SHALL exclude AWS SDK v3 packages from the bundle since Lambda provides them natively

### Requirement 3: Lambda Handler Routing and Validation

**User Story:** As a system operator, I want the Lambda function to validate all inputs and route to the correct action, so that invalid requests are rejected early and actions are dispatched predictably.

#### Acceptance Criteria

1. WHEN the Lambda_Handler receives an event, THE Lambda_Handler SHALL validate the event payload against a valibot schema before processing
2. IF the event payload fails validation, THEN THE Lambda_Handler SHALL return a structured error response with validation details
3. WHEN the Lambda_Handler receives a valid event, THE Lambda_Handler SHALL route to the correct action handler based on the `action` field
4. THE Lambda_Handler SHALL support the following action values: `scan`, `getStateUrl`, `apply`
5. THE Lambda_Handler SHALL exist as a single file containing all action routing and handler logic

### Requirement 4: Remote Scan

**User Story:** As a DevOps engineer, I want to trigger a remote scan of my AWS environment, so that the canonical state is updated on S3 without needing broad local IAM permissions.

#### Acceptance Criteria

1. WHEN the user runs `aws-accounts remote scan`, THE CLI SHALL invoke the Lambda_Handler with action `scan`
2. WHEN the Lambda_Handler receives action `scan`, THE Lambda_Handler SHALL scan AWS Organizations and IAM Identity Center using the same logic as the local scan command
3. WHEN the scan completes, THE Lambda_Handler SHALL write the resulting state to the State_Bucket as `state.json`
4. WHEN the scan completes, THE Lambda_Handler SHALL return a summary containing resource counts (OUs, accounts, users, groups, permission sets, account assignments)
5. WHEN the CLI receives the scan summary, THE CLI SHALL display the resource counts to the user
6. WHEN the scan completes successfully, THE CLI SHALL update the local State_Cache with the new state

### Requirement 5: Remote Plan with State Fetching

**User Story:** As a DevOps engineer, I want to compute a plan locally using remote state, so that I can preview changes without broad AWS permissions and without waiting for a remote round-trip for diffing.

#### Acceptance Criteria

1. WHEN the user runs `aws-accounts remote plan`, THE CLI SHALL check the local State_Cache for freshness against the configured TTL
2. WHILE the State_Cache is fresh (age less than TTL), THE CLI SHALL use the cached state for plan computation without fetching from S3
3. WHEN the State_Cache is stale or missing, THE CLI SHALL invoke the Lambda_Handler with action `getStateUrl` to obtain a Pre_Signed_URL
4. WHEN the CLI receives a Pre_Signed_URL, THE CLI SHALL download the state from S3 using an HTTPS GET request
5. WHEN the CLI downloads fresh state, THE CLI SHALL update the local State_Cache and record the fetch timestamp
6. WHEN the CLI has current state available, THE CLI SHALL load `aws.config.ts`, compute the desired state, and diff it against the current state locally using the existing diff logic
7. WHEN the plan is computed, THE CLI SHALL display the operations to the user in the same format as the local `plan` command

### Requirement 6: Remote Apply

**User Story:** As a DevOps engineer, I want to send planned operations to the Lambda for execution, so that AWS changes are applied from a controlled environment with appropriate permissions.

#### Acceptance Criteria

1. WHEN the user runs `aws-accounts remote apply`, THE CLI SHALL compute a plan using the same logic as `remote plan`
2. WHEN the plan contains operations, THE CLI SHALL display the plan and prompt the user for confirmation before proceeding
3. WHEN the user confirms, THE CLI SHALL invoke the Lambda_Handler with action `apply` and the list of operations as payload
4. WHEN the Lambda_Handler receives action `apply`, THE Lambda_Handler SHALL execute each operation sequentially using the same operation execution logic as the local apply command
5. WHEN the Lambda_Handler completes all operations, THE Lambda_Handler SHALL write the updated state to the State_Bucket
6. WHEN the Lambda_Handler completes apply, THE Lambda_Handler SHALL return the updated state in the response
7. WHEN the CLI receives the updated state from apply, THE CLI SHALL update the local State_Cache
8. IF an operation fails during apply, THEN THE Lambda_Handler SHALL stop execution, write the partially-updated state to the State_Bucket, and return an error response containing the failure details and the number of operations completed
9. WHEN the CLI receives an error response from apply, THE CLI SHALL display the failure details and update the local State_Cache with the partial state

### Requirement 7: State Caching with Configurable TTL

**User Story:** As a DevOps engineer, I want the CLI to cache remote state locally with a configurable TTL, so that repeated plan commands are fast without unnecessary network calls.

#### Acceptance Criteria

1. THE CLI SHALL store the cached state in a file alongside the existing `state.json` (e.g., `.remote-state-cache.json`)
2. THE CLI SHALL record the timestamp of the last successful state fetch alongside the cached state
3. WHEN the cached state age exceeds the TTL from the Context_File, THE CLI SHALL treat the cache as stale
4. WHEN the user runs `aws-accounts remote plan --refresh`, THE CLI SHALL bypass the cache and fetch fresh state regardless of TTL
5. THE Context_File SHALL store the TTL value in seconds under a `stateCacheTtlSeconds` field within the `deployment` key

### Requirement 8: State Locking

**User Story:** As a DevOps engineer, I want concurrent applies to be prevented, so that state corruption from parallel executions cannot occur.

#### Acceptance Criteria

1. THE Lambda_Handler SHALL use S3 conditional writes (If-None-Match or equivalent) when writing state to prevent concurrent write conflicts
2. THE Lambda function SHALL have reserved concurrency set to 1 to prevent concurrent Lambda executions
3. IF an S3 conditional write fails due to a conflict, THEN THE Lambda_Handler SHALL return an error indicating a concurrent operation is in progress
4. WHEN the CLI receives a concurrency conflict error, THE CLI SHALL display a message indicating another apply is in progress and the user should retry later

### Requirement 9: Remote Upgrade Placeholder

**User Story:** As a DevOps engineer, I want a command to update the deployed Lambda function, so that I can deploy new versions of the handler after upgrading the CLI package.

#### Acceptance Criteria

1. WHEN the user runs `aws-accounts remote upgrade`, THE CLI SHALL read the Lambda ARN from the Context_File
2. WHEN the user runs `aws-accounts remote upgrade`, THE CLI SHALL update the Lambda function code using the current `dist/lambda.zip`
3. IF the `dist/lambda.zip` file does not exist, THEN THE CLI SHALL exit with an error instructing the user to run `npm run build:lambda` first
4. WHEN the upgrade completes, THE CLI SHALL log the new Lambda version or last-modified timestamp

### Requirement 10: CLI Command Structure

**User Story:** As a DevOps engineer, I want remote commands grouped under a `remote` subcommand, so that local and remote workflows are clearly separated.

#### Acceptance Criteria

1. THE CLI SHALL register a `remote` command group with subcommands: `bootstrap`, `scan`, `plan`, `apply`, `upgrade`
2. THE CLI SHALL preserve all existing local commands (`scan`, `bootstrap`, `init`, `regenerate`, `plan`, `apply`, `graveyard`) unchanged
3. WHEN the user runs `aws-accounts remote` without a subcommand, THE CLI SHALL display help text listing available remote subcommands
4. THE CLI SHALL accept `--profile` and `--region` flags for all remote subcommands to configure AWS credentials

### Requirement 11: Lambda Handler Input/Output Schemas

**User Story:** As a developer, I want all Lambda inputs and outputs validated with valibot schemas, so that contract violations are caught at the boundary.

#### Acceptance Criteria

1. THE Lambda_Handler SHALL define a valibot schema for each action's request payload (`scan`, `getStateUrl`, `apply`)
2. THE Lambda_Handler SHALL define a valibot schema for each action's response payload
3. THE Lambda_Handler SHALL validate the response before returning it to ensure internal consistency
4. WHEN the `apply` action receives operations, THE Lambda_Handler SHALL validate each operation against the existing `operationSchema` from `operations.ts`
